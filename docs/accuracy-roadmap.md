# 検出精度改善ロードマップ

このドキュメントは、[accuracy.md](accuracy.md) の実測値（データセット `private-eval-v2`）を起点に、
現状で精度の弱いルールの根本原因を整理し、今後の改善を 6 つの角度に分けて優先順位づけした
ロードマップです。検出手法そのものの設計判断や不採用理由は
[detection-methods.md](detection-methods.md)（特に同ドキュメント「5. 将来課題」節）に、
開発・評価の運用手順は [development.md](development.md) に、後方互換・配布まわりの方針は
[roadmap.md](roadmap.md) にまとめてあります。本書はそれらと重複せず、**実測 F1/FP/FN を
起点にした改善の優先順位づけ**に範囲を絞ります。

以降、改善候補には次のいずれかの状態ラベルを付けます。

- **本ブランチで実装**: 本ロードマップの起点となった改善と同時期に対応し、実装済み（または
  実装中）のもの。
- **将来課題**: 方向性は妥当と判断したが、本ロードマップの時点では未着手のもの。
- **見送り**: 検討した結果、当面は対応しないと判断したもの（理由を併記）。

## 2. 根本原因分析

medium プロファイルで残る FP 9 件・FN 2 件は、次のごく少数の実装上の原因に集約できます。

### FP（9件）

**7件**: `internal/detect/negative_context.go` の採番ラベル隣接判定 `hasUnitBefore` は、値の
直前で許容する空白を半角スペースとタブだけに限定しています。そのため、ラベルと値の間に
次のようなものが挟まると判定が途切れ、抑制されるべき値がそのまま検出されてしまいます。

- 助詞を挟む: 「受付番号は1234…です」（「受付番号」と値の間に「は」）
- ASCII 語 + コロンを挟む: 「ジョブID: 2025…」（「ジョブ」と値の間に「ID: 」）
- コロンのみを挟む: 「型番: AB1234…」（「型番」と値の間に「: 」）

`internal/rule/negative_context.go` の `NegativeKeywordLabelPrefix` クラス
（`numberingLabelPrefixes` 語彙: 伝票番号・受付番号・予約番号・ビルド番号・シリアル番号・
ジョブ・型番・品番・図面・追跡番号）は値への直接隣接だけを見る設計で、
`NegativeKeywordGeneric` のような窓一致へのフォールバックを持ちません。影響ルールは
`jp-my-number`・`jp-postal-code`・`jp-passport` など、`digitRuleUnitAdjacentNegativeContext`
を使う数字系ルールです。

**残り2件**: `jp-phone-number` の該当パターン（区切りあり携帯・区切りなし携帯・区切りあり
固定電話・国際表記 +81）は `IgnoreNegativeContext: true` のため、`NegativeContext` の判定
自体が最初から適用されません。「電話機SKU: 090-…」「電話API version: 03-…」のような業務 ID・
バージョン表記が、負文脈の助けなしにそのまま検出されます。

### FN（2件、データセット唯一の見逃し）

- `jp-address`: 都道府県名を伴わないラベル付き住所（例:「住所: 渋谷区神南1-2-3」）は既定
  プロファイルでは非検出です。同じ入力は `jp-address-high-recall`（`--high-recall` 時のみ
  有効）でのみ検出できます。
- `jp-bank-account`: パターンが `\d{7}` の連続一致のみで、空白区切り（「口座番号 123 4…」）に
  対応していません。

## 3. 改善の6つの角度

上記の原因分析を踏まえ、改善候補を 6 つの角度に整理します。

| # | 角度 | 状態 | 概要 |
|---|---|---|---|
| 3.1 | 精度（FP削減） | 本ブランチで実装（一部見送り） | 採番ラベル隣接判定の緩和・phone負文脈のラベル限定化・postal境界ガード強化 |
| 3.2 | 再現率（FN削減） | 本ブランチで実装（一部将来課題） | 区切り表記ゆれの吸収・都道府県なしラベル付き住所の既定検出 |
| 3.3 | 辞書拡充 | 将来課題（市外局番は対応不要） | 銀行名・金融機関コード・姓辞書の段階拡張 |
| 3.4 | 新規カテゴリ | 将来課題（一部見送り） | 健康保険記号番号ペア→IMEI→和暦生年月日→IP/MAC |
| 3.5 | エンジン・正規化 | 将来課題（一部見送り） | sourceExtensions拡張・ゆうちょdiff対応 |
| 3.6 | 評価基盤 | 将来課題 | キャリブレーション床のラチェット・再現率床ゲート・合成ケース拡充 |

### 3.1 精度（FP削減）

- **本ブランチで実装**: 負文脈のグルー（ラベルと値の間に挟まる文字）許容と、採番ラベル
  接尾辞ヒューリスティックの追加。保護規則として、ラベル自体がそのルールの正文脈語を含む
  場合は抑制しない（複合ラベルで正当な検出まで落とさないため）。
- **本ブランチで実装**: `numberingLabelPrefixes` を `digitRuleNegativeContext`
  （`jp-drivers-license`・`jp-pension-number`・`jp-bank-account`・`jp-health-insurance` 用の
  語彙）にも合成し、採番ラベル接頭パターンをこれらのルールにも適用する。
- **本ブランチで実装**: `jp-phone-number` の負文脈を「隣接ラベル限定」モード化し、
  `IgnoreNegativeContext: true` に頼らず業務 ID・型番等を抑制できるようにする。
- **本ブランチで実装**: `jp-postal-code` の `\d{3}-\d{4}` パターンの境界ガード強化。
- **見送り**: 「License No: 」のような英語ラベルによる `jp-drivers-license` の FP。原因は
  採番ラベルの隣接判定ではなく、「license」という語自体が正文脈語であることに由来する本質的な
  曖昧性（ソフトウェアライセンス番号等でも一般的に使われる語のため）で、上記の隣接判定緩和では
  解消しません。上記の改善をマージ・再測定したキャリブレーション実測を見たうえで次期判断とします。

### 3.2 再現率（FN削減）

- **本ブランチで実装**: 数字系ルールの区切り表記ゆれの吸収。いずれも `RequireContext` +
  Medium 起点を維持したうえでの追加パターンです。
  - `jp-my-number`: ドット区切り・スラッシュ区切り
  - `jp-phone-number`: スラッシュ区切り・混在区切り（例:「03.1234-5678」のようなドット+
    ハイフン混在）
  - `jp-bank-account`: 空白区切り・スラッシュ区切り（全角スラッシュは `normalize.Line` が
    半角に畳み込み済みのため、半角パターンの追加だけで両方に対応できる）
  - `jp-health-insurance`: 空白区切り
  - `jp-pension-number` / `jp-drivers-license`: ドット区切り
- **本ブランチで実装**: 都道府県名を伴わないラベル付き住所を既定プロファイルでも検出できる
  ようにする（ラベル必須 + 市区町村辞書ゲート + 町字辞書昇格という、`jp-address-high-recall`
  と同型の twin 構成を既定ルール側にも用意する）。
- **将来課題**: 40 ルーンのコンテキスト窓の限界（`RequireContextWindow` を超えて離れた
  ラベルには昇格しない設計上の制約）。文スコープ窓への一般化は大規模な設計変更を伴うため、
  ここでは対応しません。
- **将来課題**: `jp-my-number` の括弧区切り（例:「マイナンバー: (1234)…」）。

### 3.3 辞書拡充

いずれも将来課題です。文脈昇格専用の辞書拡張であり、検出のゲート自体を緩めるものではないため
リスクは低めです。

- 銀行名辞書: 現行 124 件 → zengin-code 等が公開する全国銀行協会（Zengin）公式マスタ相当の
  約 1,100 件へ。
- 金融機関コード辞書: 現行 7 件 → 全量へ。
- 姓名辞書（姓）: 現行 3,843 件から段階的に拡張。拡張時は FP コーパスの週次トレンド
  （`fp-corpus-report.yml`）で新規誤検出の増加を監視する。
- **対応不要**: 市外局番辞書（`internal/dict/area_codes.txt`）は総務省「市外局番の一覧」由来の
  387 件で既に完備です。残るのは鮮度維持の自動化（郵便番号のような月次更新ワークフロー化）
  のみで、backlog 化に留めます。

### 3.4 新規カテゴリ

いずれも将来課題です。優先順は次のとおりとします。

1. 健康保険の記号・番号ペア: `jp-yucho-account` の同一行ラベルペア + 別行相関の仕組みを
   一般化する案。ただし [detection-methods.md「対象外としたもの」](detection-methods.md#対象外としたもの理由つき)
   では、健康保険証の記号・番号の自由記述は書式の標準化が弱く静的パターン化が困難と整理済みです。
   まずはゆうちょ同様に構造を把握できる限定形（同一行ラベルペア・別行相関）から実現可能性を
   検証します。
2. IMEI（15 桁）: 既存の `checksum.Luhn` を流用でき、`RequireContext` 前提で追加コストは
   小さいと見込みます。
3. 和暦生年月日の「…生」接尾形: 現行の `jp-birthdate` はラベル（「生年月日:」等）が値の前に
   あるケースのみ対応しており、値の後ろに「生」が続く接尾形は未対応です。
4. IP / MAC アドレス: [detection-methods.md「対象外としたもの」](detection-methods.md#対象外としたもの理由つき)
   は汎用 IP アドレス検出を「日本特化の範囲外」として対象外にしていますが、ここでの候補は
   high-recall 限定 + `RequireContext` 必須という狭い条件での再検討です。

**見送り**: 法人番号単独 13 桁。国税庁が公表する公開情報であり個人情報該当性が薄いため対象外と
します（インボイス登録番号の T + 13 桁は既存の `jp-invoice-number` が `checksum.CorporateNumber`
で検証済み・担当済みです）。

学籍番号・社員番号のような組織固有 ID は、形式が組織ごとに異なるため builtin ルール化せず、
[`.jp-pii.toml` の `[[rules.custom]]`](development.md#コード変更なしでルールを追加するjp-piitoml-のカスタムルール)
によるレシピで対応する方針を維持します。学籍番号の例は development.md に既にありますが、
社員番号など代表例の追加は今後の課題です。

### 3.5 エンジン・正規化

- **将来課題**: `sourceExtensions`（`internal/detect/source_context.go`）への `.txt`/`.md`/
  `.log` 追加。これらの拡張子のファイルは現在も通常どおりフルスキャン対象ですが、
  `sourceExtensions` に無いため `.env`/`.properties` 等と同じ statement 単位の key=value
  文脈抽出の対象外です。非公開評価コーパスのケースの多くが `.txt` 拡張子のファイル名で
  走ることもあり、追加による影響を非公開評価コーパスで定量してから判断します。
- **将来課題**: ゆうちょ記号番号の別行形式の diff 走査（`--staged`/`--diff`）対応。フル
  スキャンの `ScanContent` 経由は対応済みですが、`ScanDiffHunk` は未対応です。
- **見送り（明文化）**: 丸数字・漢数字の正規化を `normalize.Line` に持たせること。
  `normalize.Line` はルーン 1:1 変換（正規化後のマッチ位置がそのまま元テキストの位置になり、
  逆引きが不要になる）というアーキテクチャ上の不変条件を持っており、これを壊すと位置の
  逆引きが必要になります。この不変条件を維持するため、必要とするルール側に文字クラスを
  持たせる現行方針（住所ルールの漢数字番地対応など）を続けます。

### 3.6 評価基盤

いずれも将来課題です。

- キャリブレーション床のラチェット: 現行の受け入れ基準（High 97.5% 以上・Medium 92% 以上）を、
  本ブランチの FP 解消のマージ・再生成後の実測を見たうえで、High 0.985 / Medium 0.95 程度へ
  引き上げる。
- 再現率床ゲート（`TestRecallAcceptance` 相当）の追加。golden 再生成時に再現率の見落としが
  紛れ込むのを防ぐ。
- 合成ポジティブケースをルールあたり 10 件から 25 件へ。1 ケースの増減が F1 に与える影響
  （1 ケース ≈ 5pt）による量子化の粗さを緩和する。
- 公開合成データセットに基づく PR CI アドバイザリ評価。`$JP_PII_FIXTURES` を持たない外部
  コントリビュータでも、変更の方向性（改善か劣化か）を PR 上で確認できるようにする。
- 運用ルール: FP/FN の修正には必ず `internal/corpusv2` の hard-negative（解消系統のケース）と
  保護用ポジティブケースを同梱する。

## 4. 運用上の注意

- 検出挙動を変える PR は、`main` へのマージ後、`private-eval` ジョブが stale な golden
  （`docs/accuracy.json`）に対して fail します。そのため、`-update` によるゴールデン・
  バッジの再生成は、マージと同日中に行う運用とします。
- `internal/detect/probe_regression_test.go` の `TestProbeRegressionKnownFalseNegatives` /
  `TestProbeRegressionKnownFalsePositives` に「検出なし」「検出あり（望ましくない）」として
  記録されているケースが、ルール改善で反転（解消）した場合は、対応する `TestProbeResolvedXxx`
  への移動を修正と同一 PR で行う慣習を維持します。
