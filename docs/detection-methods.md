# 日本特化 PII の静的検出 — 調査と手法整理

このドキュメントは、リポジトリ内のテキスト（ソースコード・設定・ドキュメント・データファイル）から
**静的なアプローチ**（実行時情報や外部 API に依存しない、テキスト解析のみ）で検出できる
日本の個人情報（PII）の種類と検出手法を整理したものです。

## 1. 背景

- 個人情報保護法は個人情報の漏えい時に報告義務（同法 26 条）を課しており、
  リポジトリへの混入は漏えいインシデントに直結する。
- 特にマイナンバー（個人番号）は番号法（マイナンバー法）により利用範囲・保管が厳格に制限される
  **特定個人情報**であり、リポジトリへの混入はそれ自体が法令違反リスクとなる。
- 秘密情報（API キー等）には gitleaks 等の実績あるツールがあるが、
  **日本語圏の PII**（全角表記・和暦・日本固有の番号体系）はそれらの検出対象外。
  本ツールはその欠落を埋める。

## 2. 静的検出アプローチの分類

| # | 手法 | 説明 | 効果 |
|---|------|------|------|
| 1 | 正規表現パターン | 番号体系・書式の構造をパターン化 | 検出の基本。単体では偽陽性が多い |
| 2 | チェックディジット検証 | 公開されている検査用数字アルゴリズムで照合 | 偽陽性を大幅に削減（マイナンバーで約 1/11、Luhn で 1/10） |
| 3 | コンテキストキーワード | 同一行の周辺語（「マイナンバー」「TEL」等）で信頼度を昇格/必須化 | 桁数だけでは曖昧な番号（免許証・口座番号等）を実用精度に |
| 4 | 日本語正規化 | 全角英数字・各種ハイフン・長音記号・全角スペースの正規化 | `０９０－…` `090ー…` などの表記ゆれを吸収。日本特化の核 |
| 5 | 構造ガード | 数字境界ガード・4-4-4-4 グループ除外・予約ドメイン除外 | 長い数字列の部分一致やテスト用ダミー値の誤検出を防止 |
| 6 | allowlist | 設定ファイル（パス/正規表現/stopword）と行内マーカー `jp-pii-detector:ignore`（旧 `pii-allow` も互換対応） | 運用上の偽陽性の逃げ道を確保。CI 導入の前提条件 |

採用しなかった静的手法:

- **辞書ベースの氏名検出**（姓名辞書との照合）: 辞書サイズと偽陽性（地名・一般名詞と同形の姓）の
  バランスが難しく v1 では見送り。ラベル付き氏名（`氏名: 山田太郎`）のみ低信頼度で対応。
- **形態素解析 / NER**: 静的だが重量級の依存（MeCab、ML モデル）が必要になり、
  commit hook の起動速度・導入容易性と相反するため対象外。将来のオプション機能候補。

## 3. PII 種別ごとの検出可能性

凡例 — 検出性: ◎ 高精度で検出可能 / ○ コンテキスト併用で実用 / △ 限定的 / × 静的検出は困難

| PII 種別 | 形式 | 検出性 | 手法 | ルール ID |
|---|---|---|---|---|
| マイナンバー（個人番号） | 12 桁 | ◎ | 正規表現 + **検査用数字**（平成 26 年総務省令第 85 号 第 5 条） | `jp-my-number` |
| 電話番号 | 0X0-XXXX-XXXX 等 | ◎ | 正規表現（携帯 060/070/080/090・IP 050・固定・+81）+ 桁数検証（固定 10 桁・携帯/IP 11 桁） | `jp-phone-number` |
| メールアドレス | RFC 5322 概形 | ◎ | 正規表現 + 予約ドメイン（RFC 2606/6761）除外 | `email-address` |
| クレジットカード番号 | 13〜19 桁 | ◎ | 正規表現 + **Luhn** + ブランドプレフィックス（JCB 3528-3589 含む） | `credit-card` |
| 郵便番号 | 〒NNN-NNNN | ◎/○ | 〒マーク付きは単独検出、なしはコンテキスト必須 | `jp-postal-code` |
| 住所 | 都道府県〜番地 | ○ | 正規表現（都道府県アンカー + 市区町村 + 番地必須） | `jp-address` |
| 運転免許証番号 | 12 桁 | ○ | 正規表現 + コンテキスト必須（検査数字アルゴリズムは非公開のため不使用） | `jp-drivers-license` |
| 旅券（パスポート）番号 | 英字 2 + 数字 7 | ○ | 正規表現 + コンテキスト必須 | `jp-passport` |
| 基礎年金番号 | 4 桁-6 桁 | ○ | 正規表現 + コンテキスト必須 | `jp-pension-number` |
| 在留カード番号 | 英 2 + 数 8 + 英 2 | ○ | 正規表現 + コンテキスト必須 | `jp-residence-card` |
| 銀行口座番号 | 7 桁 | △ | 正規表現 + コンテキスト必須（構造情報が乏しく中信頼度） | `jp-bank-account` |
| 健康保険 保険者番号等 | 8 桁 | △ | 正規表現 + コンテキスト必須 | `jp-health-insurance` |
| 氏名 | 自由形式 | △ | ラベル付き（`氏名:` 等）のみ低信頼度で検出 | `person-name` |
| 生年月日 | 西暦・和暦 | △ | ラベル付き（`生年月日:` 等）のみ検出。和暦（明治〜令和）対応 | `jp-birthdate` |
| 氏名（ラベルなし） | 自由形式 | × | 静的には辞書/NER が必要 → v1 対象外 | — |

### 対象外としたもの（理由つき）

| 種別 | 理由 |
|---|---|
| 法人番号（13 桁） | 国税庁が公表する**公開情報**であり PII ではない |
| フリーダイヤル（0120/0800） | 企業の代表番号であり個人情報性が低い。ノイズ源になるため除外 |
| IP アドレス | 日本特化の範囲外。汎用ツールの守備範囲 |
| 健康保険証の記号・番号の自由記述 | 書式の標準化が弱く静的パターン化が困難 |

## 4. 各手法の詳細

### 4.1 チェックディジット検証

**マイナンバー**（総務省令で公開されたアルゴリズム）:

```
Pn = 検査用数字を除いた 11 桁のうち、末尾から n 桁目の数字
Qn = n + 1   (1 <= n <= 6)
   = n - 5   (7 <= n <= 11)
検査用数字 = 11 - (Σ Pn×Qn mod 11)   ただし mod 11 <= 1 のとき 0
```

ランダムな 12 桁が偶然通過する確率は約 9% で、パターン単体より偽陽性を 1 桁下げられる。
さらにコンテキストキーワード（「マイナンバー」「個人番号」）の有無で信頼度を medium/high に分ける。

**クレジットカード**: Luhn（ISO/IEC 7812）+ ブランドプレフィックス + 桁数の三段検証。
日本市場向けに JCB（3528–3589, 16–19 桁）を明示的にサポート。

**運転免許証番号**は検査数字を持つことが知られているがアルゴリズムが**公式に公開されていない**ため、
リバースエンジニアリング由来の実装は採用せず、コンテキスト必須の 12 桁パターンとした。

### 4.2 日本語正規化（ルーン 1:1 変換）

検出前に各行へ以下を適用する。すべて 1 文字 → 1 文字の変換に限定しているため、
正規化後のマッチ位置がそのまま元テキストの位置になり、列番号の報告が正確に保てる。

- 全角英数字・記号 → 半角（`０９０－１２３４` → `090-1234`、`＠` → `@`、`：` → `:`)
- 全角スペース → 半角スペース
- ハイフン類似 8 文字（‐ ‑ ‒ – — ― − ﹣）→ `-`
- 長音記号「ー」は**数字に隣接する場合のみ** `-` に変換（`090ー1234` は電話番号、`サーバー` は保持）

### 4.3 コンテキストキーワードと信頼度

各検出は low / medium / high の信頼度を持つ（既定の報告閾値は medium）。

- パターン + 検証だけで十分な精度のもの（〒付き郵便番号、Luhn 通過カード番号）→ 単独で high
- パターンは確実だが PII 性が文脈依存のもの（区切りなし携帯番号、マイナンバー）→ medium、
  同一行にキーワードがあれば high に昇格
- 桁数しか手がかりがないもの（免許証・口座番号など）→ キーワードが**なければ破棄**（RequireContext）。
  キーワードは検出の前提条件であり昇格の根拠にはしないため、ルールごとの基準信頼度
  （口座番号・保険者番号は medium、免許証・旅券などは high）のまま報告する

### 4.4 構造ガードによる偽陽性対策

- **数字境界ガード**: `(?:^|[^0-9])` で長い数字列（タイムスタンプ・ハッシュ等）の部分一致を防ぐ
- **4-4-4-4 グループ除外**: `1234-5678-9012-3456` の先頭 12 桁が偶然マイナンバーの
  検査用数字を通過するケースがあるため、ハイフン継続を検出して除外
- **ダミー値除外**: 全桁同一の番号。メールは RFC 2606/6761 予約ドメイン（.test、.invalid、
  .localhost、example TLD）に加え、ダミー値での使用が多い `.local`（mDNS, RFC 6762）と、
  ドメイン中の任意の位置の `example` ラベル（example.co.jp 等）も除外する
  （`example.jp` のような実在ドメインも除外される点はトレードオフ）

### 4.5 走査対象の最適化（hook / CI 向け）

- pre-commit: `git diff --staged -U0` の**追加行のみ**を走査 → コミット内容の増分だけを高速に検査
- CI（Pull Request）: `--diff origin/main...HEAD` で PR の追加行のみを走査
- フルスキャン: バイナリ判定（先頭 8KB の NUL バイト）、5MB 超のファイル除外、
  `node_modules` 等の依存ディレクトリの自動スキップ

### 4.6 出力時のマスキング

検出した PII をそのままログ（CI のコンソールや SARIF）に出力すると**二次漏えい**になるため、
既定で `09*********78` の形にマスクする。`--unmask` はローカル調査用。

## 5. 将来課題

- 姓名辞書 / 軽量 NER による氏名検出（オプション機能として）
- 複数行コンテキスト（ラベルと値が別の行にある CSV/YAML への対応）
- ベースラインファイル（既存の検出を凍結し、新規追加分のみ fail させる導入支援）
- 在留カード番号のチェックディジット（信頼できる公開仕様が確認できれば導入）
