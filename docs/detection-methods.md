# 日本特化 PII の静的検出手法の調査と整理

このドキュメントは、リポジトリ内のテキスト（ソースコード、設定、ドキュメント、データファイル）から
**静的なアプローチ**（実行時情報や外部 API に依存しない、テキスト解析のみ）で検出できる
日本の個人情報（PII）の種類と検出手法を整理したものです。

## 1. 背景

- 個人情報保護法は個人情報の漏えい時に報告義務（同法 26 条）を課しており、
  リポジトリへの混入は漏えいインシデントに直結する。
- 特にマイナンバー（個人番号）は番号法（マイナンバー法）により利用範囲や保管が厳格に制限される
  **特定個人情報**であり、リポジトリへの混入はそれ自体が法令違反リスクとなる。
- 秘密情報（API キー等）には gitleaks 等の実績あるツールがあるが、
  **日本語圏の PII**（全角表記、和暦、日本固有の番号体系）はそれらの検出対象外。
  本ツールはその欠落を埋める。

## 2. 静的検出アプローチの分類

| # | 手法 | 説明 | 効果 |
|---|------|------|------|
| 1 | 正規表現パターン | 番号体系・書式の構造をパターン化 | 検出の基本。単体では偽陽性が多い |
| 2 | チェックディジット検証 | 公開されている検査用数字アルゴリズムで照合 | 偽陽性を大幅に削減（マイナンバーで約 1/11、Luhn で 1/10） |
| 3 | コンテキストキーワード | 周辺語（「マイナンバー」「TEL」等）で信頼度を昇格/必須化。ASCII 語は単語境界つきで照合 | 桁数だけでは曖昧な番号（免許証・口座番号等）を実用精度に |
| 4 | 日本語正規化 | 全角英数字・各種ハイフン・長音記号・全角スペースの正規化 | `０９０－…` `090ー…` などの表記ゆれを吸収。日本特化の核 |
| 5 | 構造ガード | 数字境界ガード・4-4-4-4 グループ除外・予約ドメイン除外 | 長い数字列の部分一致やテスト用ダミー値の誤検出を防止 |
| 6 | ソースコード文脈 | 拡張子で主要なソースコード/設定ファイルを判定し、変数名・キー名・代入/ key-value 構造を軽量に利用 | `bankAccountNo: "1234567"` のようなコード中の値を拾い、`orderId` や `version` などの数字ノイズを抑える |
| 7 | allowlist | 設定ファイル（パス glob/正規表現、値の正規表現、stopword）と行内マーカー `jp-pii-detector:ignore`（詳細は後述） | 運用上の偽陽性の逃げ道を確保。CI 導入の前提条件 |

allowlist の行内マーカーは `jp-pii-detector:ignore`（旧 `pii-allow` も互換対応）で、
いずれもトークン境界一致で判定するため、`pii-allowlist` のような無関係な文字列には一致しない。

採用しなかった静的手法:

- **広範な辞書ベースの氏名検出**（姓名辞書で全文を走査）: 辞書サイズと偽陽性（地名や一般名詞と同形の姓）の
  バランスが難しく、見送り。代わりに、コンパクトな姓名辞書（`internal/dict`）を**検出器ではなく検証器**として使う。
  ラベルや敬称で得た氏名候補を姓名辞書で照合し、一致すれば `medium`（既定 min_confidence=medium でも
  報告）、一致しなければ収録外の人名も取りこぼさないよう `low`（既定では非表示）で拾う。弱いラベル
  （`姓:`/`名:`/`last_name:` 等）や敬称アンカーは辞書照合を必須とするため常に `medium`、`氏名:` 等の
  強いラベルは辞書照合ありの `medium` 判定と辞書照合なしの `low` 判定を同一正規表現の 2 枚組で持つ。
- **形態素解析 / NER**: 静的だが重量級の依存（MeCab、ML モデル）が必要になり、
  commit hook の起動速度や導入容易性と相反するため対象外。将来のオプション機能候補。

## 3. PII 種別ごとの検出可能性

凡例（検出性）：◎ 高精度で検出可能 / ○ コンテキスト併用で実用 / △ 限定的 / × 静的検出は困難

| PII 種別 | 形式 | 検出性 | 手法 | ルール ID |
|---|---|---|---|---|
| マイナンバー（個人番号） | 12 桁 | ◎ | 正規表現 + **検査用数字**（平成 26 年総務省令第 85 号 第 5 条） | `jp-my-number` |
| 電話番号 | 0X0-XXXX-XXXX 等 | ◎ | 正規表現（携帯 060/070/080/090・IP 050・固定・+81）+ 桁数検証（固定 10 桁・携帯/IP 11 桁）。下位種別（service/ip/mobile/fixed/international）を判定し `Reason.kind` に記録（[4.3 節](#電話番号の下位種別分類)参照） | `jp-phone-number` |
| メールアドレス | RFC 5322 概形 | ◎ | 正規表現 + IANA TLD 実在チェック + 予約ドメイン（RFC 2606/6761）除外 | `email-address` |
| クレジットカード番号 | 13〜19 桁 | ◎ | 正規表現 + **Luhn** + ブランドプレフィックス（JCB 3528-3589 含む） | `credit-card` |
| 郵便番号 | 〒NNN-NNNN | ◎/○ | 日本郵便データ由来の 7 桁完全一致（実在する 7 桁集合のビットセット照合）。〒マーク付きは単独検出、なしはコンテキスト必須 | `jp-postal-code` |
| 住所 | 都道府県〜番地 | ○ | 正規表現（都道府県アンカー + 市区町村 + 番地必須）。番地は丁目→番→号の連鎖（`2丁目10番7号` / `2-10-7` / `10番地の7`）を最後まで捕捉。高再現率モードでは都道府県なし住所も限定検出 | `jp-address` / `jp-address-high-recall` |
| 運転免許証番号 | 12 桁 | ○ | 正規表現 + コンテキスト必須（検査数字アルゴリズムは非公開のため不使用） | `jp-drivers-license` |
| 旅券（パスポート）番号 | 英字 2 + 数字 7 | ○ | 正規表現 + コンテキスト必須 | `jp-passport` |
| 基礎年金番号 | 4 桁-6 桁 | ○ | 正規表現 + コンテキスト必須 | `jp-pension-number` |
| 在留カード番号 | 英 2 + 数 8 + 英 2 | ○ | 正規表現 + コンテキスト必須 | `jp-residence-card` |
| 銀行口座番号 | 7 桁 | △ | 正規表現 + コンテキスト必須（構造情報が乏しく中信頼度） | `jp-bank-account` |
| 健康保険 保険者番号等 | 8 桁 | △ | 正規表現 + コンテキスト必須 | `jp-health-insurance` |
| 氏名 | 自由形式 | △ | ラベル付き（`氏名:`/`お名前:`/`customer_name:` 等）で検出。値が姓名辞書に一致すれば `medium`（既定でも報告）、一致しなければ `low`（既定では非表示）。プレースホルダ・非人物キーを除外。詳細は[「氏名ルールの段階設計」](#氏名ルールの段階設計)を参照 | `person-name` / `person-name-high-recall` / `person-name-structured` |
| 生年月日 | 西暦・和暦・区切りなし8桁 | △ | ラベル付き（`生年月日:`/`誕生日:`/`birthday:` 等）のみ検出。西暦・和暦（元号略記・元年表記を含む）・区切りなし8桁に対応し、実在する暦日かを検証。詳細は[「生年月日ルールの詳細」](#生年月日ルールの詳細)を参照 | `jp-birthdate` |
| 氏名（完全なラベルなし） | 自由形式 | × | 静的には辞書/NER が必要。担当者・敬称などのアンカーがない自由文は対象外 | — |

### 対象外としたもの（理由つき）

| 種別 | 理由 |
|---|---|
| 法人番号（13 桁） | 国税庁が公表する**公開情報**であり PII ではない |
| IP アドレス | 日本特化の範囲外。汎用ツールの守備範囲 |
| 健康保険証の記号・番号の自由記述 | 書式の標準化が弱く静的パターン化が困難 |

フリーダイヤル（0120/0800 等）は企業の代表番号で個人情報性が低いが、`jp-phone-number` の
区切りあり固定電話パターンは書式上どうしても検出してしまう（末尾 3〜4 桁を許容するパターンが
ナビダイヤル等の表記も拾うため）。除外ではなく**下位種別として分類した上で opt-in で除外可能**に
している。詳細は [4.3 節「電話番号の下位種別分類」](#電話番号の下位種別分類)を参照。

## 4. 各手法の詳細

### 4.1 チェックディジット検証

**マイナンバー**（総務省令で公開されたアルゴリズム）:

```
Pn = 検査用数字を除いた 11 桁のうち、末尾から n 桁目の数字
Qn = n + 1   (1 <= n <= 6)
   = n - 5   (7 <= n <= 11)
検査用数字 = 11 - (Σ Pn×Qn mod 11)   ただし mod 11 <= 1 のとき 0
```

ランダムな 12 桁が偶然通過する確率は約 9% で、パターン単体より偽陽性を 1 桁下げられる。
さらにコンテキストキーワード（「マイナンバー」「個人番号」）の有無で信頼度を medium/high に分ける。

検査用数字だけでは「先頭ゼロ埋め＋末尾昇順連番」（`0000001` 等）のダミー値を防げないため、追加で
`IsZeroPaddedSequential`（`internal/checksum`）を棄却条件に加えている。マイナンバーは日付を符号化しないため、
先頭 8 桁が実在する暦日（YYYYMMDD）に見えることだけでは棄却しない。

**クレジットカード**: Luhn（ISO/IEC 7812）+ ブランドプレフィックス + 桁数の三段検証。
日本市場向けに JCB（3528–3589, 16–19 桁）を明示的にサポート。決済処理業者が公開するテスト用 PAN
も実在番号と同じ形式・検査用数字を持ち、値だけでは安全に区別できないため検出対象とする。
あわせて、13 桁の旧 Visa 形式
（`4` 始まり・13 桁）は現在ほぼ廃止されているため検出対象から外した。稀に現存する 13 桁 Visa の
検出漏れより、`45`/`49` 始まりの JAN コード等 13 桁数字列の誤検出抑制を優先するトレードオフである。

**運転免許証番号**は検査数字を持つことが知られているがアルゴリズムが**公式に公開されていない**ため、
リバースエンジニアリング由来の実装は採用せず、コンテキスト必須の 12 桁パターンとした。全桁同一・
先頭ゼロ（公安委員会コード未満）のダミー値のみ Validate で棄却する。

**銀行口座番号・健康保険 保険者番号/被保険者番号**は検査用数字を持たないため、全桁同一の
ダミー値のみを Validate で棄却する。連番は実在しうるため棄却しない。

**旅券（パスポート）番号・在留カード番号**も検査用数字を持たない。旅券番号は末尾 7 桁が全桁同一
（`0000000` 等）の場合のみ棄却する。冊子記号の先頭文字制限（`T`/`M` 等）は外務省/ICAO の一次情報で
裏取りができるまで導入しない。在留カード番号は、出入国在留管理庁の文字集合仕様で使われない英字
`I`・`O` を含む値と、数字 8 桁が全桁同一の値を棄却する。

### 4.2 日本語正規化（ルーン 1:1 変換）

検出前に各行へ以下を適用する。すべて 1 文字 → 1 文字の変換に限定しているため、
正規化後のマッチ位置がそのまま元テキストの位置になり、列番号の報告が正確に保てる。

- 全角英数字・記号 → 半角（`１２３－４５` → `123-45`、`＠` → `@`、`：` → `:`)
- 全角スペース → 半角スペース
- ハイフン類似 8 文字（‐ ‑ ‒ – — ― − ﹣）→ `-`
- 長音記号「ー」は**数字に隣接する場合のみ** `-` に変換（`12ー34` のように数字間なら変換し、`サーバー` は保持）

### 4.3 コンテキストキーワードと信頼度

各検出は low / medium / high の信頼度を持つ（既定の報告閾値は medium）。以下、信頼度の設計と
判定の仕組みを話題ごとに整理する。

#### 信頼度の 3 段階

- パターン + 検証だけで十分な精度のもの（〒付き郵便番号、Luhn 通過カード番号）→ 単独で high
- パターンは確実だが PII 性が文脈依存のもの（区切りなし携帯番号、マイナンバー）→ medium、
  同一行にキーワードがあれば high に昇格
- 桁数しか手がかりがないもの（免許証や口座番号など）→ キーワードが**なければ破棄**（RequireContext）。
  キーワードは検出の前提条件であり昇格の根拠にはしないため、ルールごとの基準信頼度
  （口座番号と保険者番号は medium、免許証や旅券などは high）のまま報告する

#### キーワードの照合方法

- ASCII のコンテキスト語は、`tel` が `hotel` に、`card` が `discard` に含まれるような
  部分一致では成立させない。日本語の語は単語境界が曖昧なため、現時点では部分一致のまま扱う。

#### 近接ウィンドウと負コンテキスト

- 免許証番号、基礎年金番号、銀行口座番号、健康保険番号のような低エントロピーの桁ベースルールは、
  肯定キーワードがマッチ前後 40 ルーン以内にある場合のみ成立する。同じ近傍に金額、数量、連番 ID を
  示す語（`円`、`人`、`注文`、`管理番号` など）がある場合は棄却する。

#### 電話番号の下位種別分類

下位種別分類（`Rule.Kind`、internal/rule）はルール横断の汎用機構である。対応するルールが検出値を
分類して `Reason.kind`（`--explain` で確認可能。JSON 出力では `reason.kind`）に記録し、
`.jp-pii.toml` の `[rules] exclude_kinds` に種別名を列挙すると、検出可否・信頼度はそのままに
その種別の finding だけを結果から除外できる。最初にこの仕組みを使ったのは以下の電話番号の下位種別
分類で、`jp-invoice-number` の `public-business`（このセクションの末尾）も同じ機構に乗っている。

`jp-phone-number` は書式が電話番号として成立する値をすべて検出するため、0120/0800/0570/0990/0180
（フリーダイヤル・ナビダイヤル・ダイヤルQ2・テレドーム等のサービス番号）や 050（IP電話）も検出対象に
含まれる。これらは検出可否・信頼度には影響しないが、検出結果の `Reason.kind`（`--explain` で確認可能。
JSON 出力では `reason.kind`）に下位種別が記録される:

- `service`: 0120/0800/0570/0990/0180 始まり（フリーダイヤル・ナビダイヤル・ダイヤルQ2・テレドーム）
- `ip`: 050 始まり（IP電話）
- `mobile`: 060/070/080/090 始まり（携帯電話。060 は本来 FMC だが、この分類では mobile 側に含める）
- `international`: `+81` 表記で、国内形に読み替えても上記のいずれにも該当しない場合（+81 表記の固定電話等）
- `fixed`: それ以外（区切りあり/なしの固定電話）

サービス番号等をノイズとして除外したい場合は、`.jp-pii.toml` に以下を設定する（既定は
`exclude_kinds` 未設定＝全種別検出で、既存の検出結果には影響しない）:

```toml
[rules]
exclude_kinds = ["service"]
```

同じ仕組みは `jp-invoice-number`（適格請求書発行事業者登録番号 / インボイス登録番号）にも使われている。
登録番号は国税庁の適格請求書発行事業者公表サイトで誰でも検索・閲覧できる公開情報であり（3 節の
「対象外としたもの」にある法人番号と同じ性質）、マイナンバー等の機微な個人情報と同列に CI を
失敗させると運用上のノイズになりうる。そのため検出結果には常に `Reason.kind = "public-business"` が
記録され（`PublicBusinessKind`、internal/rule/identifier_kind.go）、ルールを無効化せずに下位種別として
opt-in で除外できる（電話番号側の kind とは語彙が独立しているため、`exclude_kinds` に両方を指定しない
限り互いに影響しない）:

```toml
[rules]
exclude_kinds = ["public-business"]
```

#### 複数行にまたがる検出

- `ScanContent` は行単位検出を維持したうえで、隣接 2 行の仮想ウィンドウを RequireContext ルールに
  限定して追加走査する。`口座番号:` の次行に `1234567` があるようなラベル/値分離ケースを拾い、
  検出位置は元の行と列へ戻して報告する。

#### ソースコード文脈

- `.go`、`.ts`、`.py`、`.rb`、`.java`、`.swift`、`.rs`、`.php`、`.sql`、`.json`、`.yaml`、
  `.env`、`.properties` などの主要なソースコード/設定ファイルでは、変数名・キー名・
  代入/key-value 構造を軽量な文脈として扱う。正規表現の走査対象は元の行だけで、
  合成した文脈文字列から新しい検出値は作らない。AST 解析や言語別 parser は使わず、
  `bankAccountNo = "1234567"` のような明確な statement の値範囲にだけ文脈を適用する。
- `id`、`version`、`count`、`amount` などのコード限定の負コンテキストは、同じ statement の
  値にだけ効かせ、通常の文章や diff の文脈行から追加行を抑制する用途には使わない。
- `.json`/`.yaml`/`.yml` は、上記の同一行 key=value 抽出に加えて「オブジェクト（マッピング）
  スコープ」の親キー文脈も付与する（`internal/detect/object_scope.go`）。ネストしたマッピングの
  直近の親キー（1 段のみ。祖父母チェーンは付けない — 誤帰属リスクと効果のバランスを取るための
  意図的なスコープ限定）を、値の行の PositiveText/NegativeText の両方へ CSV/SQL の列コンテキストと
  同じ方式（正負の実際の判定はルール語彙に委ねる）で追加する。例えば `phone:` の下に
  `home: 0466221111` が続く YAML では、"home" 単体では電話番号ルールの Context 語彙に一致せず
  RequireContext を満たせないが、親キー "phone" が加わることで検出できるようになる。JSON は
  ブレース/ブラケットの深さスタック、YAML はインデント幅のスタックで追跡する軽量な行ベースの状態
  機械（AST・専用パーサは使わない）で、フロー形式（`{a: 1}`）・複数行文字列（`|`/`>`）・配列項目
  （`- `）は保守的にスキップし（親を付けない）、JSON はパース不能・スタック不整合を検出した行以降も
  同様に付与を打ち切る（いずれも安全側）。フル走査限定で、diff 走査（`--staged`/`--diff`）では
  使わない（hunk はファイル冒頭からの相対位置を持たず、深さ・インデントのスタックを正しく復元
  できないため）。

#### 氏名ルールの段階設計

氏名は、ラベルが「人」をどれだけ強く示すかに応じて、段階的に信頼度の扱いを分けている。

- **強いラベル**（`氏名:`/`お名前:`/`customer_name:`/`full_name:` 等、語そのものが「人」を表すもの）は、
  同一正規表現の 2 枚組（twin）で辞書照合ありの `medium` 判定と辞書照合なしの `low` 判定を両方持つ。
  値が姓名辞書に一致すれば `medium`（既定 min_confidence=medium でも報告）、一致しない収録外の人名は
  `low` のまま取りこぼさない（`resolveOverlaps` が同一スパンで信頼度の高い `medium` を残す）。
- **弱いラベル**（`姓:`/`名:`/`last_name:`/`first_name:`）は、コンパクトな姓名辞書（`internal/dict`）で
  人名らしさを検証済みのため常に `medium`（`名:` は名、`姓:` は姓とラベル種別を区別し、日常語と衝突する
  1 文字の単独要素は棄却する）。
- **ハンドル名にもなりうるラベル**（`user_name:`/`account_name:`/裸の `name:`）は、姓+名に分割できる値
  だけ `medium`（強いラベルと同様の twin）とし、単独の姓・名一致（`name: 大和` 等、地名・一般名詞と
  同形の単独姓）は `low` のまま昇格させない。
- **ラベルの前方境界とプレースホルダ除外**: ラベルの前方境界は漢字、かな、英数字、`_` に加え、裸の
  `name:` では `-` と `.` も禁止し、`登録名前:`、`会社名:`、`project_name:`、`project.name:` のように
  ラベル語が複合キーの一部になっているケースを除外する。プレースホルダ（未定、該当なし、テスト 等。
  `未定です` のような接尾辞付きも部分一致で）の値も全ラベルで棄却する。
- **敬称アンカー（高再現率モード）**: `様`/`さん` 等の敬称アンカーは人物を強く示すため、辞書 allowlist
  ではなく組織語尾（`株式会社` や `商事` 等）の denylist で `田中商事様` 等を棄却し、辞書未収録の実在
  人名（`桐谷太郎様` 等）を巻き添えで落とさない。担当者ラベルなど人物シグナルが弱い箇所は辞書
  allowlist を使うため、コンパクト辞書が再現率の上限になる（辞書拡充は将来課題）。
- **ラベルと値の行分離（高再現率モード）**: 氏名のラベルと値が別の行に分かれるフォーム形式
  （`氏名:` の次行に `山田太郎` 等）も検出する（`person-name-structured`）。前述の `ScanContent` の
  隣接行走査で、氏名系の強いラベルだけの行とその次の値行を組み合わせ、値はコンパクト姓名辞書で照合する。
  「次行＝値」の前提は同一行ほど強くないため、同一行の強いラベル（辞書照合なし）より厳しく辞書照合を
  必須にして偽陽性を抑える（辞書未収録の氏名は取りこぼす）。`姓:`/`名:` の隣接フィールド結合は対象外
  （将来課題）。評価データセットでは `content` ケースを使うことで、この `ScanContent` 経路も評価できる
  （README バッジの既存プロファイルでは高再現率ルールは無効）。

#### 複数エンティティ共起ブースト（`cooccurrence_boost`、opt-in）

`[rules] cooccurrence_boost = true` を指定すると、氏名系ルール
（`person-name` / `person-name-high-recall`）の `low` / `medium` 候補を、
同一ファイル内の近傍に別カテゴリの検証済み高信頼 PII があるときだけ 1 段昇格（`low`→`medium`、
まれに `medium`→`high`）させる。CSV/DB ダンプや帳票テキストでは、氏名単体は誤検出が多くても、
電話番号・郵便番号・マイナンバーなどの複数種が近接して揃えば「個人情報レコード」らしさが強く、
それを信頼度に反映する機能。

- **既定は無効**（opt-in）。辞書不一致の氏名は既定で `low` のまま非表示という方針（4.3 節）を
  維持し、既存ユーザーの出力には一切影響しない。
- 判定は `ScanContent`（フルスキャン）専用。`ScanDiffHunk`（`--staged`/`--diff`）は文脈行を
  昇格の根拠にしない設計（4.1 節）を維持するため対象外。
- 昇格対象は `Validated`（チェックサム・辞書検証を通過）または `ContextKeywords` 有りの候補に限る
  （プレースホルダ値などバリデーション自体を通らない候補はそもそも候補にならないため昇格しない）。
- 昇格の根拠（アンカー）にできるのは、`High` 信頼度かつ検証済み（`Validated` またはラベル必須の
  `RequireContext`）に到達した別カテゴリの候補のみ（マイナンバー・電話番号・郵便番号・
  メールアドレス・クレジットカード・運転免許証番号・旅券番号・基礎年金番号・在留カード番号）。
  住所（`jp-address`/`jp-address-high-recall`）はチェックサム検証もラベル必須化も無く、
  試合スコアや日付を住所と誤検出するボーダーケースを道連れに昇格させるリスクが相対的に高いため、
  現時点ではアンカーに含めない。銀行口座番号・健康保険番号・生年月日は `Base` が `medium` 止まりで
  「高信頼」の基準を満たさないため同様に対象外。
- ウィンドウは同一ファイル内 ±5 行。増幅リスク（1 つの真の PII が近傍のボーダー誤検出を
  道連れに昇格させること）を抑えるため意図的に狭く取っている。
- 病的なファイル（氏名ラベルが大量に並ぶ生成ファイル等）でメモリ・処理時間が悪化しないよう、
  1 ファイルあたりの保持候補数に上限を設けている。
- `.json`/`.yaml`/`.yml` では、前述の「オブジェクト（マッピング）スコープ」（`object_scope.go`）が
  求める RecordID（JSON はトップレベル直下の各オブジェクト、YAML はトップレベルキーごとに 1
  レコード）を併用する。候補行・アンカー行の両方が RecordID を持つ場合は、同一 RecordID であることを
  唯一の昇格条件にする（距離を問わない — レコードは ±5 行より長く続くことがあるため）。これにより、
  たまたま数行以内に隣接する別オブジェクトの高信頼 PII（例: 別ユーザーの電話番号）が、無関係な
  弱い候補（例: 別ユーザーの氏名）を道連れに昇格させる事故を防ぐ一方、同一オブジェクト内であれば
  ±5 行を超えて離れていても正しく昇格できる。候補行・アンカー行のどちらか一方でも RecordID を
  持たない場合（JSON/YAML 以外のファイル、またはレコード構造の外側の行）は、その組み合わせに
  ついてのみ従来どおり ±5 行窓へフォールバックするため、既存ファイルでの挙動は変わらない。

#### 生年月日ルールの詳細

- 生年月日はラベル（`生年月日:`/`誕生日:`/`birthday:`/`date_of_birth:`/`DOB:` 等、ラベル直後の
  `(西暦)` 等の注記も許容）に限定して検出する。
- 和暦（明治〜令和、`S60.1.2` 等の単字アルファベット略記・`令和元年` 等の元年表記も対応）と、
  ラベル直結の区切りなし8桁（`19850102`）に対応する。
- 実在する暦日かを検証し、`2023-99-99`・非閏年の `2-29`・元号年範囲外・月日レンジ外の値は棄却する。

#### 高再現率モード

- `--high-recall` または `[rules] high_recall = true` を指定した場合のみ、都道府県なし住所や、
  前述の担当者・敬称アンカー付き氏名、ラベルと値が別行の氏名（フォーム形式）などの追加ルールを
  有効化する。既定では偽陽性を避けるため無効。

#### テスト経路の信頼度降格

- `testdata/`・`fixtures/`・`__tests__/`・`spec/`・`mocks/`・`seed(s)/` 配下や
  `*_test.go`/`*.spec.*`/`*.test.*` は、`RequireContext` かつ Base が medium のルール
  （口座番号・保険者番号・郵便番号の桁のみパターン）に限り Low へ 1 段階降格する
  （`Detector.ScanContent`/`ScanDiffHunk` の post-processing、`internal/detect/path_profile.go`）。
  除外ではなく降格のため `--min-confidence low` で常に確認でき、`[rules] path_demotion = false`
  で無効化できる。Base が high 固定のルール（免許証番号・クレジットカード番号・マイナンバー等）は
  対象外で、実データがテスト経路に混入した場合の検出力を落とさない。詳細は
  [`docs/development.md`](development.md) を参照。

### 4.4 構造ガードによる偽陽性対策

- **数字境界ガード**: `(?:^|[^0-9])` で長い数字列の部分一致を防ぐ。マイナンバー・電話番号・
  クレジットカード番号のように文脈なしでも拾う番号は、さらに前後の ASCII 英数字を境界扱いしない。
  マイナンバーとクレジットカード番号では前後のハイフンも境界扱いしないことで、hex ハッシュや
  UUID 内部の偶然一致を除外する
- **4-4-4-4 グループ除外**: `1234-5678-9012-3456` の先頭 12 桁が偶然マイナンバーの
  検査用数字を通過するケースがあるため、ハイフン継続を検出して除外
- **郵便番号の実在検証**: 日本郵便の UTF-8 版郵便番号データから生成した 7 桁実在集合の
  ビットセット（1,250,000 バイトを `//go:embed`）で 7 桁完全一致を判定し、`000-0000` のような
  非実在番号や、上位 3 桁は実在しても 7 桁としては未割当の番号（`150-9999` 等）を除外する。
  生成手順は `internal/dict/gen` と CLAUDE.md を参照。
- **ダミー値除外**: 全桁同一の番号。マイナンバー・運転免許証番号では、さらに「先頭ゼロ埋め＋
  末尾昇順連番」や「全体が昇順・降順の等差数列」（`0000001`・`1234567` 等）も棄却する。
  メールは IANA の TLD 一覧に存在しない TLD と
  RFC 2606/6761 予約ドメイン（.test、.invalid、
  .localhost、example TLD）に加え、ダミー値での使用が多い `.local`（mDNS, RFC 6762）と、
  ドメイン中の任意の位置の `example` ラベル（example.co.jp 等）も除外する
  （`example.jp` のような実在ドメインも除外される点はトレードオフ）。加えて、ローカル部・
  ドメイン第 1 ラベルにダミー値でよく使われる語（`hoge`・`fuga`・`dummy`・`hogehoge`・`sample`・
  `foo`・`bar`）を部分一致で含む場合も除外する（`barclays.co.jp` の `bar` のように、これらの語を
  偶然含む実在のローカル部・ドメインを巻き添えで棄却しうるトレードオフがある）

### 4.5 走査対象の最適化（hook / CI 向け）

- pre-commit: `git diff --staged -U3` で文脈行付きの差分を取得して走査。検出値が
  **追加行に乗っているもののみ**を報告（文脈行上の既存 PII は報告しない）。ラベルが
  **直前・直後の未変更行**にあり値だけを追加したケースでもコンテキスト必須ルールが発火する
  （行をまたぐ相関は隣接 ±1 行）。文脈行は正のコンテキスト補完にのみ使い、抑制には使わない
- CI（Pull Request）: `--diff origin/main...HEAD` で PR の差分を同様に走査
- フルスキャン: バイナリ判定（先頭 8KB の NUL バイト）、5MB 超のファイル除外、
  `node_modules` 等の依存ディレクトリの自動スキップ
- フルスキャンはさらに、UTF-16/Shift_JIS/EUC-JP/ISO-2022-JP からの透過的なデコード
  （疑わしければ通常のバイナリ/UTF-8 判定へフォールバック）の後段として、最終的な
  UTF-8 テキストに対し JSON の `\uXXXX` エスケープ（`json.dumps(ensure_ascii=True)` の
  出力・`.ipynb`・各種ログに頻出）を復号するビューを適用する。氏名・住所のような
  非 ASCII の値がエスケープの中に隠れて正規表現から完全にすり抜ける問題への対応で、
  復号結果が制御文字（改行等）になるエスケープは復号しないため、行番号は常に原文と
  一致する
- 同じ復号ビューの仕組みは HTML の数値文字参照（`&#23665;` や `&#x5c71;` 等）と URL の
  パーセントエンコード（`%E5%B1%B1%E7%94%B0` 等）にも拡張されており、`\uXXXX` → HTML
  数値文字参照 → パーセントエンコードの順に直列適用される（`scan --stdin` でも同じ復号
  チェーンが働く）。数百種類ある名前実体（`&amp;` 等）は構文があいまいで誤爆リスクが
  高いため対象外とし、構文が厳密な数値文字参照だけを復号する。パーセントエンコードは
  `%XX` が 2 個以上切れ目なく連続し、全体がマルチバイト文字を含む正当な UTF-8 になる
  場合だけを対象とし、`%20`（半角スペース）のような単発の ASCII 1 バイトは URL の構造
  を壊すリスクがあるため対象外とする

### 4.6 出力時のマスキング

検出した PII をそのままログ（CI のコンソールや SARIF）に出力すると**二次漏えい**になるため、
既定で `09*********XX` の形にマスクする。`--unmask` はローカル調査用。
`--explain` は text / JSON 出力に検出理由（基準信頼度、昇格に使ったコンテキスト語、
近接必須設定、検証結果など）を追加するが、検出値のマスキング方針は変えない。

### 4.7 ベースライン方式（既存検出の凍結）

既存リポジトリへの導入時、過去に混入済みの PII・ダミー値が一斉に検出され、対処手段が
`--exit-zero`（全検出無効）か広域 allowlist しかないと運用が破綻しやすい。gitleaks の
`--baseline-path` / `.gitleaksignore`、detect-secrets の `.secrets.baseline`（hashed_secret
照合）と同様に、`internal/baseline` は導入時点の既知の検出を **fingerprint（値ハッシュ）** として
ベースラインファイルに記録し、以降のスキャンでは新規追加分のみを fail させる。

- fingerprint は `hex(HMAC-SHA256(salt, ruleID + "\x00" + file + "\x00" + match))`。行番号は
  含まないため、ファイル内で行が前後に移動しても再検出されない。一方でルール ID・ファイルパス・
  検出値のいずれかが変われば別の fingerprint になり、再度検出される
- `scan --update-baseline --baseline <path>`: 現在の findings でベースラインファイルを新規作成
  （salt を自動生成）、または既存ファイルに追記し、終了コード 0 で終了する
- `scan --baseline <path>`: ベースラインに記録済みの fingerprint を持つ検出を結果・終了コードから
  除外する。`--staged` / `--diff` / フルスキャンいずれのモードとも併用できる
- `scan --baseline <path> --show-baseline`: 除外された検出も参考表示するが、終了コードには
  影響させない
- JSON 出力（`--baseline` 指定時）には各検出の `fingerprint` フィールドが付与され、
  ベースラインファイルへの手動追記など参照用途に使える

**セキュリティ上の限界**: salt 付きハッシュは複数リポジトリ間でのレインボーテーブル使い回しを
防ぐものであり、ベースラインファイル自体にアクセスできる者による低エントロピーな値
（7 桁の口座番号等）への総当たり照合を防ぐものではない。ベースラインファイルは元のソース履歴と
**同程度の機密度**で扱うこと（秘匿リポジトリでの管理、アクセス制御など）。

### 4.8 外部レコグナイザ連携（`external_recognizer`、opt-in）

軽量 NER（GiNZA/BERT 等）による氏名検出は、Go バイナリに ML ランタイムを同梱すると
commit hook としての起動速度・配布容易性（単一バイナリ）を損なう（2 節「採用しなかった静的手法」）。
`internal/external` は依存を一切足さずに、**ユーザーが指定した外部コマンド**を検出候補の
生成器として呼び出す opt-in の接続点だけを提供する。NER モデル自体はこのリポジトリに含まれず、
利用者が用意した任意のプログラム（Python + GiNZA/BERT、他言語の実装、単純な辞書引きスクリプト等）
を接続する。

#### 有効化

既定は未設定＝完全に無効。`.jp-pii.toml` に `[external_recognizer]` を追加すると有効になる:

```toml
[external_recognizer]
command = ["python3", "my_ner.py"]  # argv 配列。シェル解釈はしない
timeout_seconds = 30                # 既定 30
max_findings = 1000                 # 既定 1000
```

動くサンプル（辞書 1 語をハードコードして返すだけのデモ）が
[`integrations/external-recognizer/`](../integrations/external-recognizer/) にある。

適用範囲はフルスキャンと `scan --stdin` のみ（`--staged`/`--diff` の git diff 経路は
v1 の対象外）。1 回の走査につき子プロセスを 1 つだけ起動し、対象ファイルすべてを
まとめて渡す（後述）。

#### プロトコル v1（JSONL）

子プロセスの起動コスト（インタプリタ起動 + モデルロード）はファイル数に比例させたくないため、
「ファイルごとに子プロセスを 1 つ起動する」案ではなく、**「1 回の走査につき子プロセスを 1 つ」**
起動し、複数ファイルを JSONL でまとめて送受信する方式を採用した（実装のしやすさより実運用の
起動コストを優先したトレードオフ。詳細な比較は `internal/external` のパッケージコメントを参照）。

親→子（標準入力、1 行 1 JSON、全ファイル送信後に標準入力を閉じる）:

```jsonl
{"version":1,"file":"src/users.py","text":"<ファイル全文>"}
{"version":1,"file":"src/orders.py","text":"<ファイル全文>"}
```

子→親（標準出力、1 行 1 検出。ファイルをまたいで任意の順序でよい）:

```jsonl
{"file":"src/users.py","rule_id":"person-name-external","line":12,"column":5,"length":3,"confidence":"medium"}
```

| フィールド | 説明 |
|---|---|
| `file` | 対応するリクエスト行の `file` をそのまま返す |
| `rule_id` | **`-external` で終わる必要がある**（組み込みルール ID の偽装を拒否するため。例: `jp-my-number` を騙って送っても破棄される） |
| `line` | 1 始まりの行番号（`text` を `"\n"` で分割した行） |
| `column` | 1 始まりの**ルーン**列（UTF-8 のバイト位置ではなくコードポイント単位。internal/normalize と同じ座標系） |
| `length` | 検出値のルーン数 |
| `confidence` | `"low"` \| `"medium"` \| `"high"`。それ以外は `low` として扱う |

検出値そのもの（マッチ文字列）はプロトコルに含めない。子の自己申告値は信用せず、親側が
`file` の `text` から `line`/`column`/`length` で切り出して `Finding.Match` に入れる
（出力時のマスキングは既存の report 層がそのまま適用する）。標準エラーは子の診断用に
自由に使ってよく、レポートには出さずログ相当として標準エラーへ転記される。

#### 失敗時の扱い

検出器本体を壊さないことを最優先する。子プロセスの**タイムアウト・0 以外の終了コード・
標準出力の不正な JSON 行**は、いずれも「この走査回の外部候補をすべて破棄」として扱い、
通常の組み込みルールによる検出だけで走査を継続する（部分的な結果を信用しない。壊れた/
意図しない出力を返す子が一部だけ正しく見える結果を混ぜてくる可能性を考慮する）。

一方、個々の候補が意味的に不正な場合はその候補**だけ**を破棄する（`internal/detect` 側の
検証）:

- `rule_id` が `-external` で終わらない
- `[rules] disabled` に列挙されている
- `line`/`column`/`length` が対象ファイルのテキスト範囲外
- 値が乗る行に `jp-pii-detector:ignore` マーカーがある
- allowlist（stopword・正規表現）に一致する
- `confidence` が不正・空（この場合は破棄せず `low` として扱う）

生存した候補は、既存の **allowlist・ignore マーカー・`min_confidence`・重複解決
（`resolveOverlapsPerLine`）** をすべて通ったうえで、通常の検出結果と統合される。同一箇所に
組み込みルールの検出と外部候補が重なった場合も、外部候補だからという特別扱いはせず、
既存の信頼度・範囲優先の決着規則がそのまま適用される。`--explain` の理由表示には
「外部レコグナイザ=true」が付き、`Reason.Validated` は常に `false` のまま
（チェックサム等の内部検証を経ていないことを表す）。

#### セキュリティ上の注意

`[external_recognizer].command` は**設定ファイルに書かれた任意のコマンドを実行する**機能である。
そのため:

- **リポジトリ内の `.jp-pii.toml` を信用できない環境では使わないこと**（fork からの PR を
  未レビューで CI 実行するような構成では、悪意ある `command` が紛れ込む余地がある）。
- 可能な限り `--config <path>` で明示指定した設定ファイルでのみ有効化し、`.jp-pii.toml` の
  リポジトリルートからの自動上方探索（`config.Load` の既定動作）に外部コマンド実行の可否を
  委ねないことを推奨する。
- 環境変数や自動探索ではコマンドを拾わない（`[external_recognizer].command` を明示的に
  設定ファイルへ書いた場合のみ有効）。CI の秘密情報を経由してコマンドを差し替えられる
  経路は存在しない。
- 子プロセスは走査対象ファイルの**全文**を受け取る。子プロセス自体・その依存関係
  （PyPI パッケージ等）の信頼性は利用者の責任範囲になる。
